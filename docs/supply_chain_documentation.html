📌 1. Project Overview & Objectives

This project implements a production-grade supply chain demand forecasting system for the manufacturing sector, inspired by enterprise platforms like RELEX Solutions. It combines machine learning, a REST API, and an interactive dashboard into a complete end-to-end pipeline.

🎯 Business Objectives

Demand Forecasting

Predict weekly demand for 10 industrial SKUs one week ahead, enabling proactive production planning and inventory management.

Supply Risk Detection

Automatically detect demand spikes, drops, and supplier reliability issues before they impact production continuity.

Explainability

Provide SHAP-based explanations so supply chain managers understand why the model made each prediction and can trust the output.

Real-time Dashboard

Interactive React dashboard connecting to the ML model via FastAPI, showing forecasts, alerts, and SHAP charts updated on demand.

🏭 Sector & Scope

The simulation covers a 2-year period (104 weeks) with 10 industrial products across mechanical and electronic categories, 5 industrial clients (automotive, construction, energy, medical, aerospace), and 3 raw material suppliers from Germany, Taiwan, and Spain. All data is synthetic but designed to replicate real-world manufacturing demand patterns including seasonality, economic cycles, promotional events, and supply disruptions.

📦 5. Data Pipeline (Notebooks 01 & 02)

5.1 Synthetic Data Generation

2 years of weekly data (104 weeks) across 5 variable groups. Each group is correlated with others to replicate real supply chain interdependencies.

5.2 Feature Engineering

The 6 raw CSVs are merged into a single flat table and enriched with 70+ features.

Lag features

ordered_qty_lag_1/2/4/8/13/26/52
backlog, utilisation, capacity lags

Rolling statistics

mean/std over 4w and 12w windows
demand_momentum · trend_slope

Derived features

fill_rate · scrap_rate
supply_risk_score · week_sin/cos

Data leakage prevention

All rolling statistics use shift(1) before the window — the model never sees the current week's data when computing its features. Current-week variables (ordered_qty, backlog) are dropped; only their lagged versions are used.

🤖 6. ML Model (Notebook 03)

6.1 Algorithm: XGBoost

XGBoost builds an ensemble of decision trees sequentially, where each tree corrects the errors of the previous ones. It was chosen because it handles mixed variable types naturally, is robust to outliers, learns feature interactions automatically, trains quickly, and is fully compatible with SHAP for explainability.

6.2 Walk-Forward Validation

Why not a random train/test split?

A random split in time series allows the model to train on week 80 and predict week 30 — impossible in production. Walk-forward validation simulates real deployment: train on all past data, predict one week ahead, expand, repeat.

Fold 1:  Train [weeks 0-39]  → Predict week 40
Fold 2:  Train [weeks 0-40]  → Predict week 41
Fold 3:  Train [weeks 0-41]  → Predict week 42
...
Fold N:  Train [weeks 0-T-1] → Predict week T
Final metrics = average across all folds

6.3 Hyperparameters

ParameterValueRationale
n_estimators300Enough trees to capture complex patterns
max_depth5Moderate depth — learns interactions without overfitting
learning_rate0.05Small steps — forces gradual improvement
subsample0.880% of rows per tree — adds randomness
colsample_bytree0.880% of features per tree — prevents dominance
reg_alpha / reg_lambda0.1 / 1.0L1 + L2 regularisation to control complexity

6.4 Performance Metrics

MAPE

Mean Absolute % Error. Easy to explain to business: "off by X% on average"

RMSE

Root Mean Squared Error. Penalises large errors more than small ones

Bias

Signed mean error. Positive = overestimates. Negative = underestimates

6.5 SHAP Explainability

SHAP (SHapley Additive exPlanations) assigns each feature a contribution value for every individual prediction. Unlike global feature importance, SHAP explains why the model predicted a specific value for a specific product in a specific week — transforming XGBoost from a black box into a fully interpretable system that supply chain managers can trust and act on.

⚡ 7. FastAPI Backend (Notebook 04)

The FastAPI backend exposes the XGBoost model as a REST API. Routers handle HTTP concerns, the predictor service handles all ML logic, and Pydantic models define strict data contracts.

API Endpoints

MethodEndpointDescription
GET/Health check — confirms model is loaded
GET/forecast/latestPredictions for all 10 products using latest data
GET/forecast/{product_id}Prediction for one specific product
POST/forecast/singleCustom prediction from provided feature dict (what-if analysis)
POST/forecast/batchMultiple predictions in a single API call
GET/forecast/shap/{product_id}SHAP explanation for latest prediction
GET/alertsActive alerts with optional ?severity=HIGH filter
GET/alerts/summaryAlert counts by type and severity

Alert Logic

🔺 DEMAND_SPIKE

Predicted demand > 130% of recent average. Action: increase production order or draw from safety stock.

🔻 DEMAND_DROP

Predicted demand < 70% of recent average. Action: reduce production order to avoid overstock.

⚠️ SUPPLY_RISK

Supply risk score > 0.6 with sustained demand. Action: contact supplier proactively.

Key Design Patterns

Singleton model loading: the XGBoost model is loaded once at startup via FastAPI's lifespan context manager. This reduces response time from ~1s to ~1ms per request.

Pydantic validation: every request and response has a typed schema. Invalid data is automatically rejected with a clear error message — no manual validation code needed.

CORS middleware: configured to allow the React frontend on localhost:3000 to call the API from the browser.

🧠 9. Key Design Decisions
🌳
XGBoost over Prophet or LSTM

Prophet handles seasonality well but is limited to univariate data. LSTM requires much more data and is harder to interpret. XGBoost handles 70+ mixed variables naturally, trains in seconds, and produces SHAP values — the best tradeoff between performance, speed, and explainability for this use case.

⏱️
Walk-forward validation over random split

A random split in time series data produces optimistic metrics that don't reflect production reality. Walk-forward validation simulates every Monday morning deployment — the only way to get honest performance estimates that match what the model will actually achieve when deployed.

✖️
Multiplicative demand model

Demand factors are multiplied together rather than added. A holiday during a peak season is worse than either alone; a promotion during an economic downturn has a dampened effect. Multiplicative models capture these compound effects naturally, producing more realistic synthetic data for the model to learn from.

🔍
SHAP for explainability

In supply chain, managers need to trust model predictions before acting on them. SHAP transforms XGBoost from a black box into a transparent system — every prediction comes with a ranked list of which variables drove it up or down, making the model auditable and actionable.

⚡
Singleton model loading in FastAPI

The XGBoost model is loaded once at API startup via FastAPI's lifespan context manager and shared across all requests. This reduces response time from ~1 second (loading from disk each time) to ~1 millisecond (inference only), making the API production-ready.

🛡️
shift(1) before rolling windows

All rolling statistics are computed on lagged data to prevent data leakage — one of the most common and damaging mistakes in time series ML. Without this, the model would see the current week's demand when computing its own features, producing artificially high training scores that collapse in production.

🚀 10. Next Steps & Extensions
🗄️ PostgreSQL integration

Replace CSV files with a proper database. SQLAlchemy + Alembic for schema migrations. Enables multi-user access, audit trails, and real-time data ingestion from ERP systems.

🧠 LSTM / Temporal Fusion Transformer

Add a deep learning model alongside XGBoost and compare metrics. TFT is the current state-of-the-art for multivariate time series forecasting with mixed variable types.

🐳 Docker + cloud deployment

Containerise the FastAPI backend with Docker. Deploy to AWS/Azure/GCP. Add a CI/CD pipeline with GitHub Actions for automated model retraining when new data arrives.

🔧 Hyperparameter tuning

Use Optuna for Bayesian optimisation of XGBoost hyperparameters. Add cross-validation within each walk-forward fold for more robust and statistically sound estimates.

📊 Real ERP data

Connect to real ERP systems (SAP, Oracle) via API. Add a data validation pipeline to handle missing values, outliers, schema changes, and delayed reporting.

📧 Automated reporting

Weekly email/Slack digest of forecasts and alerts. Scheduled retraining pipeline. Model drift detection that triggers retraining when prediction errors increase over time.
